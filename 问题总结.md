# NeuPAN 真机部署问题总结

## 环境与背景

- **硬件**: MentorPi 幻尔小车（mecanum 底盘）
- **系统**: ROS 2 Humble + conda 环境 `env_neupan_ros2`
- **目标**: 在真机上运行 slam_toolbox 建图 + NeuPAN 神经网络路径规划导航

---

## 问题一：neupan_node 启动报错 `ModuleNotFoundError: No module named 'torch'`

**现象**: neupan_node 进程启动后立即退出，日志显示找不到 torch 模块。

**原因**: neupan_ros2 是 ament_python 构建的包，colcon build 生成的入口脚本 shebang 硬编码为 `#!/usr/bin/python3`，指向系统 Python，而 torch 安装在 conda 环境中。

**解决**:
1. `start.sh` 中添加 conda 环境激活：`source /home/xiaozhu/anaconda3/etc/profile.d/conda.sh && conda activate env_neupan_ros2`
2. colcon build 后用 `sed` 将入口脚本 shebang 改为 `#!/usr/bin/env python3`，使其跟随当前环境的 Python

---

## 问题二：机器人底盘不发布 TF 变换

**现象**: `ros2 topic echo /tf` 无任何输出，slam_toolbox 报 "Frame map doesn't exist"，RViz 中看不到机器人模型。

**原因**: MentorPi 小车的底层系统只发布 `/odom` 话题（`nav_msgs/Odometry`），不向 `/tf` 发布坐标变换。ROS 2 的 TF 树完全依赖 `/tf` 话题，缺少 `odom→base_footprint→base_link→lidar_frame` 变换链导致整个系统无法工作。

**解决**: 编写 `nodes/tf_publisher.py`，手动发布三条变换：
- `odom → base_footprint`：从 /odom 读取里程计数据
- `base_footprint → base_link`：固定 z=0.07 的底盘高度偏移
- `base_link → lidar_frame`：固定的激光雷达安装位置

---

## 问题三：建图与导航的里程计需求冲突

**现象**: 
- 导航启动时里程计有非零初值（机器人之前移动过），导致地图中机器人位置偏移
- 建图时使用真实里程计导致栅格地图与 SLAM 画面不重合

**原因**: 建图时需要 `odom→base_footprint = (0,0,0)` 从原点开始构建地图；导航时需要读取真实里程计来驱动机器人在 map 坐标系下移动。

**解决**: 为 `tf_publisher.py` 添加 `use_odom` 参数：
- `use_odom=false`（建图）：odom→base_footprint 发布恒等变换 `(0,0,0)`
- `use_odom=true`（导航）：订阅 /odom，记录初始值并发布相对偏移量

---

## 问题四：导航时机器人位置卡在地图原点不更新

**现象**: 导航模式下，机器人在 RViz 中始终停留在地图原点，位置和朝向都不更新，只有雷达点云在变化。

**原因**: 
1. `use_odom=true` 但 slam_toolbox localization 的 `minimum_travel_distance` 设为默认 0.2m，里程计显示无移动导致不处理扫描
2. 即使里程计有值，朝向也使用了绝对值而非相对值

**解决**:
1. 将 slam_toolbox 的 `minimum_travel_distance` 和 `minimum_travel_heading` 设为 0.0
2. 对朝向做相对化处理（见问题五）

---

## 问题五：里程计朝向未清零导致定位失败

**现象**: 导航启动后，slam_toolbox 不将当前激光扫描与已有地图对齐，而是在地图上绘制新内容，机器人位置和车头朝向完全错误。

**原因**: `odom_cb` 中对位置 `(x, y, z)` 做了清零（减去初始值），但朝向 `(qx, qy, qz, qw)` 直接使用里程计的绝对值。里程计朝向是相对于机器人开机时刻的，而地图朝向取决于建图时的初始朝向，两者不匹配导致扫描匹配失败。

**解决**: 对朝向同样做相对化处理。记录初始朝向 `q_init`，每帧计算相对旋转 `q_rel = q_init^{-1} * q_curr`（四元数共轭乘法），确保导航启动时朝向从零开始：

```python
# q_init 的共轭
iqx, iqy, iqz, iqw = -qx_i, -qy_i, -qz_i, qw_i
# q_rel = q_init^{-1} * q_curr
q_rel.x = iqw*qx_c + iqx*qw_c + iqy*qz_c - iqz*qy_c
q_rel.y = iqw*qy_c - iqx*qz_c + iqy*qw_c + iqz*qx_c
q_rel.z = iqw*qz_c + iqx*qy_c - iqy*qx_c + iqz*qw_c
q_rel.w = iqw*qw_c - iqx*qx_c - iqy*qy_c - iqz*qz_c
```

## 问题六：URDF 模型加载异常

**现象**: 尝试由 `robot_state_publisher` 单独提供 `base_footprint→base_link` 和 `base_link→lidar_frame` 变换后，模型完全损坏（轮子飞出、只剩空壳）。

**原因**: `robot_state_publisher` 发布到 `/tf_static`，而 `TransformBroadcaster` 发布到 `/tf`。两者混合使用时，static 变换以 1Hz 刷新与动态 30Hz 变换交替，导致模型闪烁甚至损坏。

**解决**: 所有三条变换统一由 `tf_publisher.py` 的 `TransformBroadcaster` 发布到 `/tf`（30Hz），不再依赖 `robot_state_publisher` 提供机器人本体变换。`robot_state_publisher` 仅保留发布 URDF 中轮子关节等非固定部件。

---

## 问题七：shebang 修复需在每次 colcon build 后执行

**现象**: colcon build 后 neupan_node 又变回 `#!/usr/bin/python3`。

**原因**: colcon build 的 ament_python 构建流程每次都会重新生成入口脚本，覆盖之前的 shebang 修改。

**解决**: 在 `start.sh` 中 colcon build 之后立即执行 `sed` 修复：
```bash
sed -i '1s|^#!/usr/bin/python3|#!/usr/bin/env python3|' \
    install/neupan_ros2/lib/neupan_ros2/neupan_node
```

## 最终架构

```
TF 变换链（tf_publisher.py, 30Hz）:
  odom → base_footprint  (来自 /odom, 位置+朝向均清零)
  base_footprint → base_link  (固定 z=0.07)
  base_link → lidar_frame  (固定安装偏移)

启动流程（start.sh）:
  1. 激活 conda 环境
  2. colcon build + shebang 修复
  3. ros2 launch neupan_navigation.launch.py
     ├── tf_publisher (use_odom=true)
     ├── robot_state_publisher (URDF 关节)
     ├── lidar 过滤节点
     ├── slam_toolbox (localization 模式, 2s 延迟)
     ├── neupan_node (4s 延迟)
     └── 启动检查节点 (6s 延迟)
```

## 关键配置文件

| 文件 | 作用 |
|------|------|
| `config/frames.yaml` | 坐标系名称、话题名称、地图名 |
| `nodes/tf_publisher.py` | TF 变换发布（核心节点） |
| `launch/neupan_navigation.launch.py` | 导航 launch 文件 |
| `launch/slam_mapping.launch.py` | 建图 launch 文件 |
| `nodes/lidar.py` | 雷达数据过滤 |
| `start.sh` | 一键启动脚本 |

---

## 操作注意事项

1. **建图前**：将机器人放在起点，运行 `bash start.sh` 启动建图 launch，用键盘或手柄遥控走完全场后保存地图
2. **导航前**：将机器人放回建图时的原点位置和朝向，运行 `bash start.sh` 启动导航 launch
3. **导航与建图的 tf_publisher 共用同一份代码**，通过 `use_odom` 参数区分行为，修改时需确保两种模式都不受影响
